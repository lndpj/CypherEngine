## 2026-04-26

### Goal

- Realign the docs with the actual codebase and the current CypherEngine direction before the filesystem and SDL3/OpenGL work begins.

### What changed

- Documented the current implemented runtime stack more honestly:
  - `common`
  - `log`
  - `platform`
  - `host`
  - `render`
  - `cmd`
  - `cvar`
  - `cfg`
- Updated phase/roadmap docs away from the old `Raylib` bootstrap note and toward:
  - `VFS`
  - `SDL3`
  - `OpenGL`
  - `Quake III BSP`
- Added API docs:
  - `docs/CYPHERENGINE_API_REFERENCE.md`
  - `docs/CYPHERENGINE_API_IMPLEMENTATION.md`
- Clarified naming:
  - `REAP` is the project/game
  - `CypherEngine` is the native engine runtime
- Simplified cfg execution flow so file loading now delegates line execution through `CypherConfig_ExecuteLine()`
- Recorded the future tooling direction:
  - small purpose-built editor
  - custom `rmdl` format
  - custom `rpk` package path when runtime pressure justifies it

### What I learned

- The project is still small enough to steer cleanly, but already large enough that stale docs cause confusion.
- Writing API docs early is worth it because subsystem intent is still easy to capture before heavy implementation drift sets in.

### Blockers / risks

- No filesystem seam exists yet, so config/assets still rely on direct file opens.
- The renderer contract exists, but there is still no actual visible platform/render backend behind it.

### Next step

- Build the `fs` subsystem next and route cfg loading through it before beginning the `SDL3` bootstrap.
